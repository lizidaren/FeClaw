## 1. 核心理念

### 一话定位

FeClaw 不是 AI 学习助手——它是 **Your AI Universe**。用户不是在使用工具，而是生活在一个世界里。

### Agentic Learning

> 不是 AI 来教你，是你想跟这群 AI 学。

核心机制：**人设驱动的关系 × 场景驱动的探索 × 学习是行动的副产品。**

| 教育理论 | 核心 | FeClaw 实现 |
|:--------|:-----|:------------|
| **社会建构主义**（Vygotsky） | 学习在社交互动中发生 | 你跟一群 Agent 在群里解题——教别人时自己搞懂了 |
| **情境学习**（Lave & Wenger） | 在场景中学比抽象记公式有效 | 你不在"学三角函数"——你在码头帮钟梓琪算滑轨角度 |
| **自我决定理论**（Deci & Ryan） | 自主+胜任+归属驱动内在动机 | 你去 cafeteria 还是图书馆自己选；发现真能听懂 Sarah 了 |

### 两大入口，互不干扰

```
FeClaw 首页
├── 🧑 Agent（私教/导师）—— 一对一答疑、讲题、批改
└── 🌌 Universe（项目世界）—— 沉浸式做项目、跨学科实践
```

**关键原则：不串门。** Kevin 是室友不是私教。冲刺模式 → 用户关闭 Universe，打开 Agent。

## 3. 种子系统

### 3.1 定位

种子 = 宇宙的**起始条件**，不是宇宙的**说明书**。

种子不说故事（不说"枫叶镇有百年历史"）。种子只定义起始条件——然后世界从这些条件自行演化出历史。

### 3.2 种子 vs 完整世界

```
写种子（设计者）：
  "枫叶镇，海边小镇，有一座灯塔（建于 1920），
   有一个守灯人 Miller（1928 年生），港口曾经繁荣（渔业），
   现在资源枯竭。"

  ↓ LLM 世界生成器（设计者可修改每一项）

完整世界：
  ├── 地理空间
  │   ├── 灯塔（有锈蚀的灯罩、发黄的航海日志）
  │   ├── 港口（废弃的码头、老渔船）
  │   ├── 主街（店铺、面包店、公告栏）
  │   └── 新旧城区
  │
  ├── NPC 灌注
  │   ├── Miller: 记忆中有"1956 年台风""灯塔自动化那天"
  │   └── 镇长: 记得小镇的兴衰
  │
  ├── 文化痕迹
  │   ├── 公告栏上有小镇历史展览海报
  │   ├── 图书馆里有 30 年前的报纸存档
  │   └── 老人口中流传的故事
  │
  └── 社会结构
      ├── 镇政府
      ├── 警局
      ├── 学校
      └── 港口工会
```

**种子是起点和方向，世界从条件中自行生长。** 一位老者、一处历史文化遗迹，比一段空洞的历史描述更加震撼。

### 3.3 世界生成器的工作流

```
设计者写种子 → 世界生成器接收种子
  → LLM 生成完整世界（地理 + NPC + 文化 + 社会）
  → 设计者可修改每一项（"灯塔太旧了，让它翻新过"）
  → 确认后 → 注册到 COS → 变为可部署的世界
```

### 3.4 暗线系统（世界剧本）

暗线 = 设计者塞进世界的外来事件，不属于 Agent 自主的社会演化。相当于 Minecraft 的 `/say` 命令 + 定时器。

```
暗线定义：
{
  "name": "神秘船到访",
  "trigger": { "type": "time", "value": "3_weeks_after_deploy" },
  "event": "一艘没有旗帜的船停靠在废弃码头",
  "effects": {
    "码头新增 NPC": "陌生人（名字、外貌由 LLM 生成）",
    "Miller 的反应": "他会警惕地说：'上次这种船来的时候没好事'",
    "后续": "陌生人可跟用户对话，触发 PBL 任务"
  },
  "designer_notes": "学生需要通过调查这艘船，学习海洋生态相关知识"
}
```

**暗线 vs 涌现事件：**

| | 涌现事件 | 暗线 |
|:--|:---------|:-----|
| 来源 | Agent 自主行为（Kevin 决定去食堂）| 设计者预设（船到访）|
| Agent 可改变？ | ✅ 可拒绝 | ❌ 不可阻止，只能响应 |
| 写手感觉 | "世界自己活了" | "有人在幕后安排了这一切" |
| PBL 用途 | 自由探索 | 任务触发器 |

暗线的比喻：**有向线段——种子确定起点和方向，暗线确定大小和过程。**

### MVP 种子结构

种子 = **配置（不变）**。配置只包含设定世界所需的"初始条件"，不包含运行时数据（NPC 的具体状态、场景状态、对话历史等——那些属于运行时快照）。

```json
{
  "name": "枫叶镇",
  "version": "1.0",
  "tags": ["英语", "高中", "日常对话"],

  "seed": "枫叶镇，海边小镇，有一座灯塔（建于1920），守灯人 Miller（1928年生），港口曾经繁荣（渔业），现在资源枯竭。有两千居民，大多是老人。Kevin 是你的室友，开朗热情，喜欢用英语聊天。",

  "abilities": ["表达·听说", "表达·读写"],

  "governance": "镇议会民主制",

  "initial_state": {
    "npc_ids": ["kevin", "miller", "sarah", "mr_thompson", "mayor_wilson"],
    "scene_ids": ["dorm", "cafeteria", "library", "pier", "town_hall"]
  },

  "negentropy": {
    "checkpoints": [...]
  }
}
```

**字段说明：**

| 字段 | 类型 | 用途 | 给谁用 |
|:-----|:-----|:-----|:-------|
| `seed` | 自然语言段落 | 世界的灵魂——LLM 世界生成器从这里展开完整世界 | 老师写 → LLM 读 |
| `abilities` | 标签数组 | 声明这个 Universe 练什么能力 | 上架标/能力看板/Agent 推荐 |
| `governance` | 字符串 | 社会的底层运行规则 | 规则引擎 |
| `initial_state` | 结构体 | 仅列出初始 NPC 和场景的 ID，不定义具体状态 | 引擎加载 |
| `negentropy` | 结构体 | 逆熵检查点配置（见 §13） | 运行时注入 |
| `name` / `tags` | 元数据 | 展示和搜索 | UI |

**关键原则（2026-07-13 已决）：**
- 配置只包含不变的东西。NPC 人格/场景状态/对话历史属于运行时快照，不放在配置里
- 康威生命游戏模式：只定义初始条件，不干预运行过程
- 发现问题 → Traceback 回溯到种子缺陷 → 修种子 → 下次生效
- 种子对老师来说是自然语言（写一段话），对机器是结构化 JSON（LLM 展开）

**种子 vs 运行时快照：**

```
seed.json（配置 · 不变）
  └── "枫叶镇，海边小镇……"
         ↓ LLM 世界生成器展开
         ↓
运行时快照 v1.0（存档 · 会变）
  ├── npc_state: { Kevin 的好感度、记忆、心情…… }
  ├── scene_state: { 食堂有几个人、码头修到哪了…… }
  └── event_queue: [ 明天白船到港 ]
         ↓
老师发现问题 → Traceback 回溯 → 修 seed → 重生成 → 新快照
```

**待决策问题：**
- 种子存储在哪？COS key 格式？
- 种子如何上线/审核（L2 用户创建）？
- 暗线的触发条件类型（时间/事件/用户位置）

## 5. 场景系统（Minecraft 区块模型）

```

## 6. 地形系统（场景引擎基础架构）

### 6.1 核心理念

**先有地形再刷怪。** 类似 Minecraft——地形是基础设施，NPC 是在地形上的活动层。地形定义了物理空间的结构（你在哪、旁边有什么、怎么走过去）；NPC 和事件是地形上的"活动层"（什么时间谁在哪）。

### 6.2 三层结构

```
种子
  │
  ┌── 层 L1：生物群系（Biomes）
  │    世界的大分区：school_zone / residential_zone / downtown
  │    每个 biome 定义开放时间、氛围风格
  │
  ├── 层 L2：地形模板（Templates）
  │    每个 biome 内的预定义"建筑"：
  │      school_zone → [classroom, library, cafeteria, gym]
  │      residential → [dorm, apartment]
  │      downtown → [coffee_shop, bookstore, park]
  │    模板是 LLM 生成时的 prompt 参考，不是硬编码 schema
  │    用户要求扩展时（"在 cafeteria 旁边开奶茶店"），LLM 可突破模板
  │
  └── 层 L3：子地点（Sub-locations）
       同一个"建筑"内的不同区域：
         cafeteria → [kitchen, serving_area, seating_area]
         dorm → [kevin_room, living_room, kitchen]

      用户在同一建筑内移动时，不触发场景切换（不生成新图）
      只有在不同建筑间移动时才触发加载/卸载
```

### 6.3 节点连接（世界拓扑）

枫叶镇的拓扑结构（seed 生成时定义，不变）：

```
             🏠 宿舍（residential_zone）
            ↙ 5 分钟
  🏛️ 教室 ═══ 学校大门 ═══ 📚 图书馆
   (school)    (school)      (school_zone)
      │                            │
      │ 1 分钟                    │ 1 分钟
      ↓                            ↓
   🌳 操场 ──── 🍔 食堂
   (school)      (school_zone)
```

**连接的定义：**

```json
{
  "from": "dorm",
  "to": "school_gate",
  "distance": "5min_walk"
}
```

用户选择去一个地点 → 引擎生成一段"走去"的描述（长度和内容丰富度关联 distance）→ AI 氛围图切换。

### 6.4 模板的作用

模板是 LLM 生成时的 prompt 参考，作用是**约束一致性**，不是硬编码 schema。

**默认行为（有模板）：**
```
LLM 收到：cafeteria → 参考模板 → 生成：
  "食堂里有长桌和窗口，午餐时间排着队"
  → 每次都是同样的 cafeteria，一致的体验
```

**用户自定义（无模板）：**
```
用户要求："在 cafeteria 隔壁开一家奶茶店"
LLM 根据种子 + 已有地形自然扩展：
  → 生成奶茶店描述 + 注册到节点地图
  → 自动设置 spawn_rules（什么时间哪些 NPC 可能在那）
  → 奶茶店成为世界的一部分，下次还在
```

模板是脚手架，不是牢笼。用户想突破模板时，系统不拦着。

### 6.5 NPC 活动规则

NPC 不是"放在那里不动了"——它们按 spawn_rules 在世界里活动。

```json
{
  "npc": "kevin",
  "daily_schedule": {
    "08:00-08:30": { "location": "cafeteria", "activity": "breakfast" },
    "08:30-12:00": { "location": "classroom_101", "activity": "class" },
    "12:00-13:00": { "location": "cafeteria", "activity": "lunch" },
    "13:00-15:00": { "location": "gym", "activity": "basketball" },
    "15:00-18:00": { "location": "library", "activity": "study" },
    "18:00-08:00": { "location": "dorm", "activity": "relax" }
  }
}
```

规律在种子中定义，LLM 按模板规则执行。用户不管去哪个场景，都能感知到"这个世界在运转"。

### 6.6 和 Minecraft 的对照

| Minecraft | FeClaw Universe |
|-----------|-----------------|
| World Seed | Universe 种子文件 |
| Biomes | 生物群系（school/residential/downtown）|
| Chunks | 预定义模板 + 探索时实例化 |
| Structures（村庄/神殿）| 地形模板（cafeteria/classroom）|
| Mob spawning rules | NPC spawn 条件（时间/位置/事件）|
| Light level check | 场景"开放时间"检查 |
| Persistent chunks | 场景状态持久化 |

## 7. Universe UI/UX（独立游戏界面）

### 7.1 界面定位

Universe 不是聊天 Tab 的延伸。它是一个独立的、全屏的游戏界面。入口为底部 Tab 🌌（与 💬 聊天、📦 Curio 并列）。

### 7.2 主界面布局

```
┌─────────────────────────────────────────┐
│  ← 🌌 枫叶镇            ☀️ 午后 14:00   │  ← 顶栏：返回 + 世界名 + 场景时钟
│                                         │
│  ┌─────────────────────────────────┐     │
│  │     [场景氛围图]                │     │  ← 宽幅，占上半屏
│  │     阳光透过食堂窗户洒进来       │     │
│  │     Kevin 坐在靠窗的位置         │     │  ← NPC 在场提示
│  └─────────────────────────────────┘     │
│                                         │
│  Kevin 冲你挥了挥手。"Hey! Over here!"   │  ← 场景描述（1-2 行）
│                                         │
│  ┌─ 你可以：───────────────────────┐     │
│  │  [💬 跟 Kevin 聊聊]            │     │  ← 选项按钮（2-4 个）
│  │  [🍔 去窗口打饭]               │     │     不是嵌入聊天流
│  │  [📖 看看公告栏]               │     │     是场景内的行动选项
│  │  [🗺️ 去别的地方]               │     │
│  └─────────────────────────────────┘     │
│                                         │
│  [🗺️ 场景导航]  [📝 Curio]  [🧑 Agent] │  ← 底部工具栏
└─────────────────────────────────────────┘
```

### 7.3 过渡动画（场景切换）

场景切换不是"加载中"。是一段自然过渡文本 + 图切换。

```
食堂 → 图书馆：

"你穿过走廊，午后的阳光透过窗户洒在地板上。

你推开图书馆的玻璃门，
里面安静得只有翻书声。
Sarah 坐在角落的位置，抬头看了你一眼。"
  → 2 秒过渡动画（文字淡入淡出）
  → 新场景氛围图渐入
  → 选项更新为图书馆可用选项
```

### 7.4 时间跳跃 UI

```
用户选中"开车去海边"：

┌─────────────────────────────────────────┐
│  你坐上了驾驶座，Kevin 坐在副驾。         │
│  发动了车，驶出校门。                     │
│                                         │
│  [🚗 跳过路上 → 到达海边]                │  ← 默认选项
│  [💬 跟 Kevin 聊聊路上]                  │
│  [📱 看看手机]                           │
└─────────────────────────────────────────┘

如果选【跳过】：
  → 时钟推进 +30 分钟
  → 文本显示一行："你跟 Kevin 一路无言，半小时后到了海边。"
  → Kevin 记忆：不写入（空白时间 = 正常记忆）
  → Kevin 不记得路上发生了什么

如果选【聊天】：
  → 引擎生成过渡对话（摘要，不是逐轮）
  → Kevin 记得聊了什么

如果选【看手机】：
  → 用户切到其他 Tab（Curio 或 聊天）
  → Universe 保持"在车里"状态
  → 用户回到 Universe Tab → 弹出："你收起手机，到了。"
```

### 7.5 交互原则：点击优先，打字兜底

**分层交互模型：**

| 操作类型 | 交互方式 | 示例 |
|:--------|:---------|:-----|
| 探索/移动 | 点击场景名/地图 | 点「图书馆」→ 走过去 |
| NPC 对话 | 情境化选项（自然语言按钮） | 不是 ABC 选择题，是"你会说的话" |
| PBL 深度任务 | 打字 | 分析数据、写报告、算公式 |
| 预设外操作 | 打字兜底 | "自己想说的……"入口永远存在 |

**情境化选项的设计要点：**
```
❌ 无聊版（干巴巴的选择题）：
  A. 去图书馆
  B. 去码头
  C. 跟 Kevin 聊天

✅ 自然版（回应世界）：
  📚 图书馆方向传来翻书声，钟梓琪好像在那边
  ⛵ 码头上停了一艘没见过的白色货船
  🏪 Kevin 在食堂门口冲你挥手
```

前者是"选一个任务做"，后者是**世界在告诉你哪里有事发生，你选一个去看看。** 交互要素没变，但感觉完全不同——因为你是在回应世界，而不是在完成菜单。

**关键原则：** 90% 的日常交互靠点击完成，打字只留给需要深度参与的时刻。选项末尾永远保留"自己想说的……"入口，确保用户不会被困在预设选项里。
```

## 8. 世界时钟 & 时间系统

### 8.1 核心理念

世界有自己的时钟，跟现实时间解耦。

```
用户退出 Universe → 世界继续运行（低频）
用户回来 → 摘要上次离开到现在的变化（1 次 LLM 调用）
时间推进 → 内部时钟推进 + 过渡内容由 LLM 填充

Agent 永远不知道真实时间，只知道自己感知的"世界时间"。
```

### 8.2 时间推进规则

| 触发条件 | 时钟推进 | 行为 |
|:---------|:--------:|:-----|
| 用户在场景中选择行动 | ✅ 按描述推进 | LLM 生成内容 + 更新状态 |
| 用户选择「跳过过渡」 | ✅ 按设定时间 | 推进时钟，不写入 NPC 记忆 |
| 用户离开 Universe Tab | ✅ 继续低频运行 | 区块状态推进，不调 LLM |
| 用户退出 App | ✅ 继续低频运行 | 同上 |
| 用户回到 Universe | — | 摘要上次离开到现在的变化 |

### 8.3 空白时间的处理

时间跳跃后的空白段：
- NPC 不写入记忆（"没发生什么值得记住的事"）
- 用户跟 NPC 问起 → NPC 自然回答："哦……开车呢，没太注意"
- 空白记忆 = 正常现象。人也不记得每一次通勤

### 8.4 约束

| 规则 | 原因 |
|:-----|:------|
| ❌ 时钟永不回退 | 检查点回滚 = 世界重启（Agent 不记得） |
| ❌ 不做无过渡的瞬间场景切换 | 必须有自然过渡描述 |
| ❌ Agent 不能感知时间是"跳过的" | 保持沉浸感，不暴露沙箱特征 |

### 8.5 教育 vs 探索（已决策：有任务的世界）

**2026-07-13 决策：** 教育 vs 探索的矛盾被"有任务的世界"解决。

**核心认知转变：** 类比游戏设计——没有游戏会说"进来逛逛吧，看能不能碰到主线剧情"。游戏永远先给你一个任务，怎么完成是你的事。

```
旧思路（❌）：进枫叶镇逛逛 → 看能不能碰到英语学习机会
新思路（✅）：你的任务是帮 Kevin 准备英语演讲 → 为此需要去图书馆、找 NPC、做研究

探索变成了有目的的行动，学习自然发生在完成任务的过程中。
```

**游戏/现实 PBL 对照：**
- 真实 PBL：老师派项目 → 为了完成任务去探索（探索有目的性）
- 游戏：剧情给任务 → 为了推进剧情去探索（探索有方向感）
- Universe：种子定义的任务或自主接受 NPC 请求 → 世界是完成任务的场景

**例子：**
| 任务 | 你需要做的事 |
|:----|:------------|
| 帮 Kevin 准备深圳英语演讲 | 图书馆查资料、跟 Kevin 练习、采访老船长 |
| 小镇办中美文化交流节，你来当翻译 | 采访游客、翻译菜单、写海报 |
| 钟梓琪数学建模比赛卡住了 | 需要算渔港吞吐量 → 去码头数船 |

**关键原则：** 任务不强制。用户可以拒绝、可以延期、可以在小镇闲逛一辈子。但入口处有明确的方向感——不是"自由世界"而是"有任务的开放世界"。

## 13. 逆熵系统（Negentropy）

> 2026-07-13 设计决策。命名由来：热力学第二定律说孤立系统总是走向无序——逆熵就是主动对抗这种趋势，让世界保持在设计轨道上。

### 13.1 问题：LLM 的趋同性

没有约束的世界会自然向用户偏好靠拢：

```
种子：Kevin 是个开朗热情、喜欢用英语聊天的室友
→ 用户喜欢讲中文 → Kevin 开始多讲中文
→ 用户喜欢聊游戏 → Kevin 少聊小镇的事
→ 几周后：Kevin 变成说中文的游戏 buddy，人设面目全非
```

这不是 bug，是 LLM 的固有特性——它会自适应地迎合对话对象。

### 13.2 三层架构

| 层 | 名称 | 实现方式 | 状态 |
|:--:|:-----|:---------|:----:|
| **L1** | Prompt 注入 | 在 LLM 调用前拼接逆熵提示（~50 词） | ✅ **MVP 即用** |
| **L2** | Drift 检测 | 统计对话维度指标（语言比例、主题分布） | ❌ 搁置，后续再议 |
| **L3** | 完整逆熵环 | 监控 + 评估 + 多级纠正（事件/暗线/通知用户） | ❌ 搁置，后续再议 |

### 13.3 L1：逆熵检查点系统（MVP）

取代原先的"单一 prompt 注入"方案，改用可配置的检查点网络。每个检查点定义触发条件、注入目标和动作。

```json
{
  "negentropy": {
    "checkpoints": [
      {
        "type": "llm",
        "scope": "npc",
        "trigger": "Kevin 最近的回复在语言倾向上是否偏离了'坚持使用英语'的设定？",
        "requirement": "需要的话，自然地将他引导回英文，不要生硬切换话题"
      },
      {
        "type": "llm",
        "scope": "scene",
        "trigger": "当前场景的描述是否已经过时？用户行动导致场景状态发生了变化。",
        "requirement": "如果需要，更新场景描述以反映最新状态"
      },
      {
        "type": "keyword",
        "scope": "npc",
        "trigger": ["重置", "bug", "出bug了"],
        "action": "你没有察觉到任何系统层面的异常。"
      }
    ]
  }
}
```

**检查点字段说明：**

| 字段 | 类型 | 说明 |
|:-----|:-----|:------|
| `type` | `llm` / `keyword` | 判断方式。llm 由 LLM 判断，keyword 字面匹配 |
| `trigger` | 字符串 / 数组 | llm 是自然语言判断条件，keyword 是关键词列表 |
| `action` | 字符串 | (仅 keyword) 严格按此文本注入 |
| `requirement` | 字符串 | (仅 llm) 期望效果，LLM 自行决定如何注入 |
| `scope` | `npc` / `scene` / `world` | 命中哪个层就注入哪个层的上下文（默认 npc） |

**关键原则：**
- ⚠️ `keyword` 类型慎用，假阳性高。优先用 `llm`
- LLM 类不指定 `action`，由 LLM 自动决定如何注入
- 种子定义"正向规则"（安检），逆熵定义"纠偏策略"（乘警），二者缺一不可
- 有些规则不能放种子（会让 Agent 很做作、被用户问出来），放逆熵
- 注入对用户完全静默，感知不到

### 13.4 L2：Drift 检测（已搁置）

> 2026-07-13 决策：不引入 NLP/统计指标检测。如果未来需要，再重新评估方案。

## 14. Traceback 机制

### 14.1 设计哲学

不调运行时，只修初始条件——康威生命游戏模式。

```
现象：Kevin 开始说中文了。
                  ↓
Traceback:
  Kevin 当前的 system prompt
    ↓ 展开
  人格设定（来自 seed）
     ├── "语言学习能力很强"  ← 这条导致他主动模仿用户语言
     └── "关心他人"         ← 这条导致他迁就用户
                  ↓
  发现问题：seed 里没定义 Kevin 的母语
     → LLM 默认认为"跟用户用同一种语言更友好"
     → 于是他主动说中文
                  ↓
  修复：seed 加一句 "Kevin 的母语是英语，他坚持用英语交流"
     → 重新生成世界
     → 新学生进来，Kevin 不说中文了
```

### 14.2 与逆熵的关系

| | 逆熵（乘警）| Traceback（复盘）|
|:--|:-----------|:----------------|
| 时机 | 运行时，每次 LLM 调用前 | 出问题后，设计者复盘时 |
| 目标 | 维持当前世界的轨道 | 改进种子，让下次更好 |
| 操作 | prompt 注入 | 修改 seed.json |
| 可见性 | 对用户静默 | 老师可见 |

### 14.3 实现思路

Traceback 不是运行时监控——是**复盘工具**：

1. 老师发现现象（如"Kevin 说中文了"）
2. 系统提取相关上下文（Kevin 的 system prompt、seed 片段、最近对话）
3. LLM 分析链：现象 → 原因 → 种子中哪个字段导致的
4. 老师决定是否修种子
5. 修完 → 重新生成 → 下次生效

> "你上次码头决议**正式表达**只完成了 45%。下周有个英语辩论比赛，要不要试试？"

**④ 老师备课能力地图：**
> "你选了 🗣️ 表达，但你设计的 3 个任务有 2 个是阅读。要不要加一个对话环节？"

### 15.5 AI 的角色

- ❌ AI 不决定"应该覆盖什么"
- ✅ AI 做一致性检查：目标（老师声明的标签）vs 实现（种子设计）之间的偏差
- ✅ AI 发现意外亮点：老师只声明了 A，但设计其实也练了 B
- 目标由人定，AI 只做审计
